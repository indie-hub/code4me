00Where v0.6 stands

v0.6.1 is a complete, end-to-end implementable framework. Conversation, Light, Standard, and Critical workflows are wired. The orchestrator contract is slim and stable; the playbook holds elaboration; references load on demand. Declarative context_queries: per agent drive context-pack assembly. The Codex shims (codex-architect, codex-developer) cover the architect-pair and one developer role. PreToolUse hooks enforce protected tests and Conversation-Mode forbidden conditions at runtime. The probe suite exercises classification, team composition, auto-escalation, and the external-agent path.

The roadmap item that matters most — live-test Light, Standard, and Critical end-to-end against real tasks — is also the one this document doesn't try to replace. Everything below is structural; the empirical validation has to happen against real work.

What this plan adds: ergonomics, observability, documentation, and a coherent vendor-pairing policy so the cross-vendor dialectic the framework already runs at the architect tier extends to every producer/verifier boundary that benefits from it.

01Part 1 — Improvements

Six categories, each with concrete actions and a rough priority (P1 = ship in v0.7 alongside live-testing; P2 = v0.7 if time, otherwise v0.8; P3 = nice to have).

Category A — Ergonomics

A1 P1

Add explicit slash commands. Right now invocation is entirely natural-language-trigger driven via the SKILL.md description. That has zero setup friction but no muscle memory and no audit handle. Add commands that wrap the most common operations:

  • /code4me-classify <task description> — runs intake + classification only, returns weight + auto-escalation + suggested team without dispatching
  • /code4me-dispatch <weight> <task> — bypasses intake, declares the weight explicitly; auto-escalation still applies
  • /code4me-status [milestone_id] — read-only status snapshot of .code4me/
  • /code4me-promote-or-revert <task_id> — closes the Conversation-Mode loop
  • /code4me-probe-run [subdir] — runs the probe suite programmatically (see B1)
  • /code4me-audit [--since=YYYY-MM-DD] — wraps bin/code4me-audit-dispatch-log

Each command is a thin file under commands/ in the plugin; the natural-language trigger continues to work alongside. The win is explicit invocation, discoverable autocomplete, and a place to hang programmatic operations (probes, audits) that don't fit the SKILL.md trigger shape.

A2 P1

Ship a /code4me-init command and starter templates. The .code4me-skeleton/ handles runtime artifacts but the first-project setup still requires manual work — project CLAUDE.md, .mcp.json, .claude/settings.json with hooks pre-wired. Add:

  • templates/CLAUDE.md.example — annotated starter with the multi-language hierarchy you already document
  • templates/.mcp-recommended.json — opinionated starter (sequential-thinking, GitHub, Playwright, filesystem) with comments per entry
  • templates/.claude/settings.json.example — hooks pre-wired with the absolute-path placeholder
  • /code4me-init — copies the templates, substitutes the plugin path, leaves placeholders where the user must edit

The README correctly explains why the plugin ships zero MCPs (project-shape, not plugin-shape). The starter template doesn't violate that — it's a recommendation file, not a configuration, and users edit it.

A3 P2

Move trigger phrases out of the SKILL.md description. The description string in skills/code4me/SKILL.md currently carries ~500 characters of trigger phrases. That string is loaded on every Claude Code session as part of skill discovery. The triggers are exactly what the model uses to decide whether to activate the skill, so this is a real trade-off, not pure waste. Action: run skill-creator's improve_description.py against a set of probe-style activation prompts; if shorter description maintains activation precision, ship the shorter one and move the long trigger list to references/triggers.md. Measure before changing.

Category B — Observability & evals

B1 P1

Make probes programmatic. Paste-and-eye-compare works for one author; it doesn't scale to "did the v0.7 prompts move anything." Build a probe runner that:

  • Reads each probes/**/*.md, parses the Input prompt and Expected blocks
  • Spins up a fresh Claude Code session via the Agent SDK or claude CLI in headless mode
  • Captures the orchestrator's transparency announcement
  • Runs an LLM-as-judge (Sonnet, one-shot prompt) to score match against the Expected block on three axes: kind, weight, team composition
  • Writes probes/results-{YYYY-MM-DD-HHMM}.jsonl with {probe, pass, score, divergence_notes}

The runner ships as bin/code4me-probe-run alongside code4me-audit-dispatch-log. Wire it to /code4me-probe-run from A1.

B2 P2

Regression budget for probe runs. Once probes are programmatic, add a configurable threshold to the runner: --max-flips=3 exits non-zero if more than three probes move from their baseline. This is the inverse of the "100% pass = evals too easy" rule — when the framework changes, a few probes should move; you halt the release only when too many do. The threshold lives in probes/budget.toml alongside the suite.

B3 P2

Extend the dispatch-log audit with deviation-pattern detection. The current code4me-audit-dispatch-log summarises counts, weight distribution, vendor split, model deviation, and auto-escalation triggers. Add:

  • Deviation-pattern surfacing — when the same role+weight combination deviates from default model on more than N% of dispatches, surface as "the default may need tuning"
  • Vendor cost rollup — sum estimated cost per vendor based on dispatched models, surface in the summary
  • Weight × outcome heatmap — markdown table of dispatches by weight against outcome (COMPLETE / BLOCKED / FORBIDDEN / TEST_QUESTION / ARCHITECTURE_BLOCK / REWORK) so you can spot which weights are noisy
  • Circuit-breaker firing log — sorted by milestone, with the trigger and the decision
B4 P3

Trace-reading discipline as a documented practice. Hamel Husain's highest-ROI evals advice is "spend 30 minutes reading 20–50 traces every meaningful change." Add a one-page docs/trace-review.md with a small checklist (transparency announcement matches dispatched team? subagent return outcomes typed correctly? context queries resolved or skipped with reason? hook fire matched intent?) so the practice is named and not just implicit.

Category C — Documentation

C1 P2

Split the README into Diataxis quadrants. 430 lines of single-document reference is correct for an internal project; if you ever shop this to others it becomes the friction point. Split into:

  • docs/tutorial-first-milestone.md — 10-minute path from install to closing a Conversation Mode task
  • docs/howto-install-hooks.md, docs/howto-configure-lsp.md, docs/howto-add-a-language.md, docs/howto-enable-codex-shims.md
  • docs/reference-*.md — the current README content, by section
  • docs/explanation-workflow-weights.md, docs/explanation-co-approval.md, docs/explanation-orchestrator-pattern.md — the why, separated from the what

The root README becomes ~80 lines: what it is, install, link to tutorial, link to the four quadrants. gstack's README does this well.

C2 P1

Introduce OpenWolf at the top of the README. OpenWolf is referenced throughout (.wolf/cerebrum.md, .wolf/anatomy.md, .wolf/buglog.json) without ever being introduced. A two-sentence intro at the top: "OpenWolf is an external persistent-knowledge system this plugin integrates with when present. The plugin works fully without it; if installed, the orchestrator reads cerebrum first to avoid re-litigating decisions you've already made." Outside readers currently can't parse half the references in tooling.md without this.

C3 P3

Add a CONTRIBUTING-style note on probe baselines. The probes README says baselines are gitignored personal artifacts. Document explicitly: where they live, the naming convention, when to capture a new one, what counts as "intentional probe movement vs. regression." Right now this is implicit in the author's head.

Category D — Architectural

D1 P2

Add provenance to the context_queries resolved output. When the orchestrator assembles a Context Pack, the resolution should record which artifact (filename + commit SHA when known) provided each item. The dispatch log gains context_provenance: [...]. This lets the audit tool answer "when this dispatch went wrong, what context was in the pack?" without re-running the workflow. Inexpensive; high diagnostic value.

D2 P2

Drop the v0.5 fallback path. The playbook already calls this out as a v0.7 deprecation. By v0.7 every shipped agent declares context_queries:, so the fallback can go. Removes the conditional branch in the dispatch protocol and simplifies the contract.

D3 P3

AGENTS.md emission. The plugin's content (conventions, tooling preferences, language guidance) is portable in principle. A small emitter that flattens the relevant plugin guidance into a project-root AGENTS.md at /code4me-init time would let Codex CLI, Gemini CLI, Aider, Continue, etc. read the same conventions when working alongside Claude Code. Low priority and the right choice only if you actually run multi-agent setups; otherwise the duplication isn't worth it.

Category E — Workflow extensions

E1 P2

Spec Kit interop. If a project has used GitHub's Spec Kit to produce a spec.md and plan.md, the orchestrator should accept those as inputs equivalent to a Milestone Spec + Tech Spec. Concretely: at intake, check for specs/<feature>/spec.md and specs/<feature>/plan.md; if present, skip Product Coach / Lead Architect for the parts already covered, and dispatch Challenger Architect (or codex-architect) for the soundness review instead. Keeps Spec Kit's strong upstream artifact pipeline; code4me handles execution.

E2 P3

Playwright MCP as a recommended verifier tool. Visual verification is one of the highest-ROI additions for any UI-touching work — 60%+ debugging time reduction per the field guide's research. Add Playwright MCP to the recommended starter .mcp.json (A2), and add a Verification-subagent directive: when the milestone touches files under known UI paths, run the smoke check through Playwright if available, capture a screenshot to .code4me/screenshots/, and reference it in the verification report.

Category F — Test & safety hardening

F1 P2

A third hook: write-allowlist for Critical work. The two existing hooks ask-gate on protected tests and forbidden conditions. A third, check-critical-write-allowlist.sh, would ask-gate any Write/Edit to paths not in .code4me/critical-allowlist.txt when a Critical milestone is active. The orchestrator writes the allowlist at Critical-mode dispatch; it deletes at task close. This is the same defense-in-depth pattern as the existing two, scoped to the highest-risk weight tier where blast radius is a real concern.

F2 P3

"Sanity probes" for production agents. Distinct from the regression probes: a small set of probes that runs before a Critical dispatch and confirms the framework is in a sane state — .code4me/ exists, hooks are installed, dispatch-log is writable, LSP is reachable for project file types, Codex CLI is present if codex-* shims are in the team. Run automatically by the orchestrator; fail fast with a clear pre-flight error.

02Part 2 — Cross-vendor team extension

Today, the cross-vendor dialectic exists at one boundary: Lead Architect (Claude) vs. Challenger Architect (Codex). The Co-Approval Rule forces both to return approved: true. That's the most valuable application of multi-vendor — same-vendor architects can converge on similar blind spots, so a different distribution genuinely catches different things.

Your request: extend the principle. One vendor produces, the other vendor verifies — and vice versa. The framework should treat producer/verifier pairs the same way it treats Lead/Challenger pairs: by default, run them on different vendors when both shims exist.

This section specifies the pairing principle, which roles need new shims, the dispatch policy, schema changes, a worked example, cost and failure-mode posture, and the probes that should verify the alternation fires correctly.

The alternation rule

The cross-vendor alternation rule

When both vendors have shims for a producer-verifier pair, the orchestrator dispatches them on opposite vendors by default. The pairing is computed at team composition time, recorded in the dispatch log under vendor_pairing, and surfaced in the Team Transparency announcement.

Cross-vendor pairing is opt-in per milestone, like the existing Codex shims. A user enables it at intake (or via /code4me-dispatch --cross-vendor); the orchestrator otherwise sticks with Claude-on-both-sides. Auto-escalated work to Critical retains the user's existing cross-vendor preference rather than forcing it on.

Producer / verifier graph (what pairs with what)

Every role in code4me either produces an artifact, verifies one, or both. The cross-vendor question is which boundaries gain enough value from a different-distribution verifier to justify the shim and the dispatch cost.

Which shims to add

Three tiers based on the value column above.

Tier 1 — High value, add in v0.7

  • codex-code-reviewer — shim of code-reviewer (Standard quality review). Modes: review-diff (default), review-files, review-spec-fit. This is the linchpin of "Claude implements, Codex reviews" and "Codex implements, Claude reviews."
  • codex-spec-to-test — shim of spec-to-test. Modes: generate (default — Test Spec + protected tests from a Tech Spec), review-test-spec (read-only soundness pass). Critical because the developer implementing against a same-vendor Test Spec can unconsciously match the test shape; a different-vendor Test Spec breaks that loop.
  • codex-security-reviewer — shim of security-reviewer. Modes: diff-focused (default), comprehensive. Fires on the same auto-escalation symptom classes. Security is where cross-vendor pays off most because OWASP categories are known to have model-dependent recall.

Tier 2 — Useful, add in v0.8

  • codex-verification — shim of verification. Modes: suite-run (default — run the full suite, report results), ac-coverage (acceptance-criteria coverage assessment against the Tech Spec). The shim's value is checking that suite-passing means the same thing across vendors; the Claude-side may have produced the implementation, so a Codex-side verifier closes the loop.
  • codex-lead-architect — shim of lead-architect. Modes: propose (default — produce Tech Spec + EDP), amend. Adding this lets you invert the architect pairing: Codex-Lead / Claude-Challenger. Useful when you want the Codex perspective to drive a milestone and Claude to pressure-test.

Tier 3 — Defer or skip

  • codex-qa — defer to v0.9. QA's value is exploratory creativity; vendor diversity helps but the Claude QA is already capable. Reassess after live-testing.
  • codex-researcher — defer. Different vendors having different sources is real value, but it's a different shape of use (you'd want to dispatch both as parallel researchers and merge, not alternate them).
  • codex-combined-reviewer — skip. Conversation/Light is meant to be cheap; cross-vendor here erodes the cost advantage that defines those weights.
  • codex-doc-writer, codex-product-coach — skip. Low blast radius, low cross-vendor value.

The vendor-pairing policy

A new references/cross-vendor-policy.md codifies the rules. The orchestrator reads it at team composition time, after team-templates but before the first dispatch.

# cross-vendor-policy.md  (excerpt)

## Alternation rule

When cross-vendor pairing is enabled for the milestone AND both shims
exist for a producer/verifier pair, dispatch them on opposite vendors.

## Pair definitions (producer → verifier)

- lead-architect ↔ challenger-architect    (Co-Approval; existing)
- spec-to-test   → developer                (test author ≠ implementer)
- developer      → code-reviewer            (implementer ≠ reviewer)
- developer      → verification             (implementer ≠ verifier)
- developer      → security-reviewer        (Critical / auto-escalated only)

## Resolution algorithm

1. If cross-vendor is OFF: every role defaults to Claude. Done.
2. If ON:
   a. Pick the Producer's vendor (default: Claude unless cerebrum or user
      override says otherwise for this milestone).
   b. For each pair the Producer participates in, set the Verifier's
      vendor to the opposite.
   c. If a transitive conflict arises (rare — e.g., spec-to-test producer
      is on vendor X, developer is on opposite Y, but code-reviewer needs
      to be opposite of developer = X, and X is also Test author of the
      tests the developer just satisfied), prefer the Producer↔Verifier
      pairing closest to the artifact under review.
   d. If a required shim is unavailable (CLI missing, key missing,
      vendor opted out), fall back to Claude-on-both-sides and record
      pairing_degraded with the reason.

## Recording

Every dispatch log entry gains:
- vendor_pairing.policy: "alternation" | "single-vendor" | "user-override"
- vendor_pairing.pair_role: which pair this dispatch fulfilled
- vendor_pairing.degraded: optional reason if fallback applied

Vendor-aware model tiers

The orchestrator currently picks Claude models per dispatch from a weight-tier heuristic — Haiku for mechanical work, Sonnet for default reasoning, Opus for architects and Critical. For Codex it has only a single user-configured default that the shim passes as --model. That's the gap: at v0.6 the orchestrator can't choose between GPT-5.4-mini, GPT-5.4, and GPT-5.5 the way it chooses between Haiku, Sonnet, and Opus.

The fix is to make model selection vendor-aware via a tier abstraction. The orchestrator picks a tier per (subagent, weight); the tier resolves to a concrete model identifier based on the vendor the pairing layer assigned. Same heuristic for both vendors; different concrete models.

The three tiers

TierUseClaudeCodex (OpenAI)
lowmechanical work, fast iteration, parallel opsHaiku 4.5GPT-5.4-mini
middefault reasoning, most workflowsSonnet 4.6GPT-5.4
higharchitecture, Critical Mode, deep reasoningOpus 4.6/4.7GPT-5.5 (or gpt-5.3-codex for code-heavy work)

Concrete model identifiers live in a single config file. The framework references tiers; users override the resolution per project without touching plugin code or agent files. When OpenAI or Anthropic ship a new model in a tier, you edit one line.

Two new config files

The first is the vendor → tier → model map. Ships with sensible defaults; project-local overrides at .code4me/vendor-models.local.yaml win when present.

# references/vendor-models.yaml

anthropic:
  low:  claude-haiku-4-5-20251001
  mid:  claude-sonnet-4-6
  high: claude-opus-4-6

openai:
  low:  gpt-5.4-mini
  mid:  gpt-5.4
  high: gpt-5.5
  # alternate "high" for code-heavy work — opt in per-role if desired:
  # high_code: gpt-5.3-codex

The second extracts the per-(subagent, weight) tier defaults from the current prose in references/model-selection.md into a structured form the orchestrator can evaluate without parsing English.

# references/model-selection.yaml  (extracted from existing prose)

defaults:
  lead-architect:        { standard: high, critical: high }
  challenger-architect:  { standard: high, critical: high }
  spec-to-test:          { standard: mid,  critical: mid  }
  developer:             { conversation: low, light: mid, standard: mid, critical: high }
  verification:          { standard: mid,  critical: mid  }
  code-reviewer:         { standard: mid,  critical: high }
  combined-reviewer:     { conversation: low, light: mid }
  qa:                    { standard: mid,  critical: mid  }
  security-reviewer:     { standard: mid,  critical: high }
  doc-writer:            { standard: low,  critical: mid  }
  researcher:            { any: mid }
  product-coach:         { any: mid }

hard_floors:
  - "architect roles never run at tier 'low'"
  - "Critical Mode never runs at tier 'low'"
  - "auto-escalated work: tier never downgraded below resolved value"
  - "cross-vendor pairing does not relax tier floors —
     a Codex developer at Critical is still tier 'high'"

Resolution algorithm

At dispatch time the orchestrator runs:

def resolve_model(role, weight, vendor, mode_context):
    # 1. Look up tier from the structured defaults
    tier = model_selection.defaults[role].get(weight) \
        or model_selection.defaults[role].get("any")

    # 2. Apply hard floors (may upgrade tier)
    tier = apply_hard_floors(role, weight, tier, mode_context)

    # 3. Apply deviation rules — complexity surprise, mid-flight escalation,
    #    previous failed dispatch at this default
    tier = apply_deviation_rules(role, weight, tier, mode_context)

    # 4. Resolve to concrete model for the chosen vendor
    return vendor_models[vendor][tier]

The hard floors are the same vendor-agnostic statements the framework already enforces — they're just expressed in tier vocabulary now instead of being tied to "never Haiku" or "never below Sonnet."

Deviation rules (vendor-agnostic)

Same three deviation rules from model-selection.md apply per vendor:

  • Bump tier up when complexity surprises you, when stakes change mid-flight, or when a previous dispatch at the default failed.
  • Record deviations in the dispatch log as tier_deviated_from_default: true with default_tier and actual_tier. Persistent deviation patterns are signal that the defaults table needs tuning.
  • Never downgrade below hard floors regardless of vendor.

Transparency announcement (updated format)

Team Transparency lines change from (anthropic:opus) to (claude:high) — vendor and tier, not vendor and concrete model. The resolved model identifier goes in the dispatch log where it's auditable; the announcement stays compact and decision-relevant.

Old:

Team for M03-T07-DEV: lead-architect (anthropic:opus),
challenger-architect (anthropic:opus), developer (anthropic:sonnet), ...

New:

Team for M07-T03-DEV (Critical, cross-vendor enabled):
lead-architect (claude:high), codex-architect (codex:high),
codex-spec-to-test (codex:mid), developer (claude:mid),
codex-verification (codex:mid), codex-code-reviewer (codex:mid),
qa (claude:mid), codex-security-reviewer (codex:high),
doc-writer (claude:low).

Rationale: tier carries the cost-and-capability signal humans use to sanity-check a dispatch. The exact model SHA is an audit-time concern, not a transparency-line concern.

What this gives you

  • Codex shims become first-class peers — picked per dispatch with the same heuristic as Claude models, not a single global default.
  • Cost predictability across vendors — same-weight work on either vendor pulls the analogous tier; rollups by tier are directly comparable.
  • Single point of update when models change — edit vendor-models.yaml, not 14 agent files.
  • Future vendor additions (Gemini, local models, anything) require only a new column in vendor-models.yaml; no logic change in the orchestrator or agents.
  • The dispatch-log analytics (B3) gain a clean (vendor × tier) grouping.

Worked example — a Critical milestone with full pairing

User enables cross-vendor for milestone M07 (a payment-flow change — Critical, auto-escalation fires on "payments" symptom class).

The orchestrator's transparency announcement before the first dispatch:

Team for M07-T03-DEV (Critical, cross-vendor enabled):

claude:high lead-architect · codex:high codex-architect (challenge mode) · codex:mid codex-spec-to-test · claude:mid developer · codex:mid codex-verification · codex:mid codex-code-reviewer · claude:mid qa · codex:high codex-security-reviewer · claude:low doc-writer

Pairing: Architect Co-Approval (Claude/Codex). Spec author (Codex) ≠ implementer (Claude). Implementer (Claude) ≠ reviewer/verifier/security (Codex). QA and docs single-vendor (no shim). What this protects against, concretely:

  • The developer (Claude) can't unconsciously match implementation to test-shape biases of the same vendor — tests come from Codex.
  • The reviewer (Codex) doesn't share blind spots with the implementer — different distribution for catching subtle bugs.
  • The security pass (Codex) sees the OWASP/STRIDE surface through different training data than the developer who wrote the auth code.
  • The architect pair retains its existing Co-Approval cross-vendor rule.

Cost & latency posture

Cross-vendor doesn't multiply dispatch count — it splits dispatches across two providers. Total Claude API tokens drop; total Codex API tokens appear. The net cost change depends on relative pricing and the size of the role's typical context pack.

  • Latency: Codex shim invocation uses codex exec with timeouts (300s arch, 600s dev). For verifiers, set lower timeouts: 240s for code-reviewer, 180s for security-reviewer, 360s for verification (suite-run can be slow). Surface as constants in references/cross-vendor-policy.md.
  • Cost: the dispatch-log audit (B3) already covers vendor cost rollup; with the pairing in place, you'll have empirical per-vendor numbers per milestone. Use the first three Critical milestones with cross-vendor on to establish a baseline.
  • Opt-in is the default. Cross-vendor is per-milestone, not project-wide. The README must say this loudly — defaulting to cross-vendor on Critical only is a defensible alternative if you find the cost worth it.

Failure-mode handling

The framework already handles per-shim failures (CLI missing, key missing, timeout, malformed response) via BLOCKED with typed blocker_type. The pairing layer adds three new conditions:

  • Producer's required verifier shim is unavailable. Action: fall back to same-vendor verifier and record pairing_degraded: shim_unavailable. Do not block the milestone — the verifier exists on the same vendor; the cross-vendor benefit is degraded but not lost.
  • Pairing conflict can't be resolved. Action: prefer the closer producer-verifier pair (e.g., developer↔code-reviewer wins over spec-to-test↔developer when both compete) and record the choice. Surface in the transparency announcement.
  • User declares "always Claude" mid-milestone. Action: honour the override for remaining dispatches; record pairing_degraded: user_override on each subsequent dispatch so the audit trail is clean.

Probes to add

Five new probes under probes/cross-vendor/ exercise the pairing logic. Each follows the existing probe shape (Input prompt → Expected transparency announcement).

  • 01-pairing-fires-on-standard.md — Standard milestone with cross-vendor enabled; Expected: developer Claude, code-reviewer Codex, verification Codex, qa Claude.
  • 02-pairing-disabled-keeps-single-vendor.md — same Standard work without cross-vendor; Expected: all Claude.
  • 03-pairing-degrades-when-shim-missing.md — Standard with cross-vendor but codex-code-reviewer unavailable; Expected: pairing_degraded recorded, fall back to claude code-reviewer.
  • 04-pairing-on-critical-auto-escalated.md — Conversation Mode request that auto-escalates to Critical because of auth touch; Expected: cross-vendor not forced on (user hadn't enabled it), security-reviewer fires single-vendor, INSIGHT recommends enabling cross-vendor for future similar work.
  • 05-spec-to-test-alternates-with-developer.md — Standard with cross-vendor; Expected: codex-spec-to-test produces tests, claude developer implements (verifies the alternation is announced, not the inverse).

What the existing shims need (small migrations)

The existing codex-architect and codex-developer shims need minor updates to participate in the pairing layer:

  • Add cross_vendor_pair_with frontmatter (challenger pairs with lead; codex-developer participates in the same producer/verifier pairs as developer).
  • Accept the new dispatch contract fields (vendor_pairing) and pass them through to the structured response.
  • Record vendor_pairing in the existing return payload so the dispatch log captures it consistently.

No behavioural change in the shims themselves — purely metadata propagation.

04What I'd explicitly not do

A short list of plausible-sounding ideas that I think would be net-negative for code4me's current shape. Worth saying explicitly because they tend to come up.

  • Don't ship MCPs as part of the plugin. The current posture — MCPs are project-shape, not plugin-shape — is correct. Resist drift. The starter .mcp.json in A2 is a recommendation file, not configuration.
  • Don't run both vendors in parallel as verifiers. "Claude code-reviewer AND Codex code-reviewer review every diff" sounds rigorous but doubles cost without commensurate benefit. The alternation rule already provides the cross-vendor coverage; parallel verification is theatre.
  • Don't force cross-vendor on Critical. Tempting because Critical is high-stakes, but the user's existing per-milestone opt-in posture is correct — the cost surface is real and reasonable people will opt out for some Critical work.
  • Don't add a Gemini shim now. Codex is the second vendor because Codex CLI exists and is mature. Adding Gemini would triple the matrix and the cost without a clear value increment. Reassess in v1.0 if Gemini CLI matures meaningfully.
  • Don't replace the natural-language SKILL.md trigger with slash-command-only invocation. Adding slash commands (A1) augments the trigger; it shouldn't supplant it. The natural-language trigger is what catches "build a feature for X" requests that the user typed without thinking about the framework. Keep both.
  • Don't merge codex-developer's review-diff mode into codex-code-reviewer. The duplication is intentional: in review-diff mode, codex-developer reviews as a developer would (with the implementer's lens); codex-code-reviewer reviews as a code reviewer (with the quality-only lens). They're not the same review even with the same input diff.
  • Don't add a "vibe coding" Conversation+ mode. Conversation Mode plus auto-escalation already covers the "small reversible change" case correctly. A mode that's looser than Conversation is just unprotected work.
